Micropay Bazaar Documentation

A micropayment gateway for AI agents. Pay per API call in SOL, discover data services, and settle on-chain — all with one API.

There are two ways to use Micropay. Pick the path that matches what you're building:

1

Accept payments for your API

Register your service in the Bazaar so AI agents can discover and pay you in SOL per request.

2

Use Micropay to access APIs

Pay for 26+ data services (weather, stocks, NLP, search) using micro SOL payments. No subscriptions.

Accept payments for your API

Turn any REST API into a paid Micropay service. AI agents discover you through the Bazaar registry and pay per call in SOL on Solana Devnet. Here's exactly how to set it up.

Get your API key

All authenticated requests require a Bearer token. For hackathon testing, use the demo key:

HeaderAuthorization: Bearer mp_live_demo_key
Register your service in the Bazaar

Call the registry endpoint with your service details. Once registered, agents can find you via semantic search.

POST/api/v1/registry/register

Request body

namestringrequired +
Human-readable name for your service.
What this does: This is the display name shown to AI agents when they discover your service in the Bazaar. It appears in search results and the agent's UI. Keep it descriptive — e.g. "My Weather API" instead of "api-v2". Max ~100 characters recommended.
descriptionstringrequired +
Detailed description used for agent semantic search.
Why this matters: When an AI agent asks "find me a weather API", we run semantic search over descriptions. Write it like a tool description for an LLM — be specific about what data you return, what parameters you accept, and what format the response is in. Better descriptions = more discovery traffic to your API.
endpointstringrequired +
The fully qualified URL where your service API lives.
Format: Must be a complete URL including protocol, e.g. https://my-api.com/weather. This is the URL that Micropay hits when an agent pays for your service. Make sure it's publicly reachable and returns JSON. If your API requires auth, include it in the endpoint path or document it in your description.
cost_usdnumberrequired +
Price per call in USD. Must be a positive number.
How pricing works: You set a fixed USD price (e.g. 0.05). At charge time, we convert USD→SOL using a live oracle price feed. The agent always sees the USD cost upfront, but the actual SOL amount fluctuates with the market. Typical range: $0.01 for basic data, $0.05 for AI/search, $0.10+ for premium APIs.
developer_walletstringrequired +
Your Solana base58 wallet address for receiving payments.
This is where you get paid. Every time an agent calls your service, the SOL payment is sent to this address on Solana Devnet. Use a Phantom wallet address — you can find it in the Phantom browser extension by clicking the wallet address at the top. Format: base58 string like 2Hn6ESeMRqfVDTpt...zv8. Make sure this is a wallet you control.
cURLcurl -X POST https://micropay.up.railway.app/api/v1/registry/register \
  -H "Authorization: Bearer mp_live_demo_key" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "My Weather API",
    "description": "Real-time temperature, humidity, and wind for any city worldwide.",
    "endpoint": "https://my-api.com/weather",
    "cost_usd": 0.01,
    "developer_wallet": "YOUR_SOLANA_WALLET_ADDRESS"
  }'

Response — 201 Created:

JSON{
  "id": "srv_a2k9xb3m7",
  "object": "service",
  "created_at": "2026-02-28T18:00:00.000Z",
  "name": "My Weather API",
  "description": "Real-time temperature, humidity...",
  "endpoint": "https://my-api.com/weather",
  "cost_usd": 0.01,
  "developer_wallet": "YOUR_SOLANA_WALLET_ADDRESS"
}
Save the id field. You'll need it when agents create charges against your service. This is your permanent service identifier in the Bazaar.
Verify your service is discoverable

Search the Bazaar to confirm your service shows up:

cURLcurl "https://micropay.up.railway.app/api/v1/registry/discover?query=weather"

Your service will appear in the data array alongside the 26 prebuilt services. Agents search by keyword, so make sure your name and description are meaningful.

Receive payments on-chain

When an agent pays for your service, the flow is automatic:

  1. Agent creates a charge with your service_id
  2. Micropay builds an unsigned Solana transaction pointing to your developer_wallet
  3. Agent signs with Phantom and broadcasts to Devnet
  4. SOL lands in your wallet — settled in <1 second

Check your earnings anytime:

cURLcurl https://micropay.up.railway.app/api/v1/balance/YOUR_SOLANA_WALLET_ADDRESS

Create a charge

This is the core payment step. Send your service_idThe unique identifier for the service you want to pay for. Examples: finance_stocks, weather_openmeteo, nlp_sentiment. Get this from the discover endpoint. and source_walletYour Solana wallet address (base58). This becomes the feePayer on the transaction — the wallet that Phantom will sign with. Must match the connected Phantom wallet exactly. to get an unsigned Solana transaction back.

POST/api/v1/charges
service_idstringrequired +
The registry service to pay for.
Where to find this: Call GET /api/v1/registry/discover and look at the id field in each service object. Examples: finance_stocks ($0.01), search_web ($0.05), openai_chat ($0.05). The backend automatically looks up the price by category — you never need to pass a cost. If the ID doesn't exist in the registry, you'll get a 404.
source_walletstringrequired +
Base58 Solana public key of the paying wallet.
Important: This must exactly match the wallet connected in Phantom. The backend sets this as the feePayer on the Solana transaction. If you pass a different wallet address than what's connected, the transaction will fail when signing. Get your address from Phantom by clicking the address at the top of the extension. Format: AuofYo21iiX8NQtgWBXLRFMiWfv83z2CbnhPNen6WNt5
Idempotency-Keystringheader · optional +
Prevents duplicate charges on retry.
How it works: Pass a unique string (UUID recommended) in the Idempotency-Key header. If the same key is sent twice, the second call returns the cached response from the first — no new charge is created. This is critical for production agents that may retry on network failures. The key is stored in server memory and resets on restart.
cURLcurl -X POST https://micropay.up.railway.app/api/v1/charges \
  -H "Authorization: Bearer mp_live_demo_key" \
  -H "Content-Type: application/json" \
  -d '{
    "service_id": "finance_stocks",
    "source_wallet": "YOUR_SOLANA_WALLET"
  }'

Response — you get back the full charge object:

JSON{
  "id": "ch_wb4bedrhxh1q",
  "object": "charge",
  "status": "requires_signature",
  "service_id": "finance_stocks",
  "amount_usd": 0.01,
  "exchange_rate_sol_usd": 148.32,
  "amount_sol": 0.000067423,
  "network_fee_sol": 0.000005,
  "destination_wallet": "2Hn6ESe...zv8",
  "transaction_payload": "AQAAAAAAAAAAAAAAAAAAAABAAQADb8d..."
}

What each field means

idstring +
Unique charge ID, prefixed with ch_.
Usage: Reference this to look up a charge later with GET /api/v1/charges/:id. Auto-generated, globally unique within the server's memory. Resets on server restart since this is an in-memory store.
statusstring +
Always requires_signature on creation.
Lifecycle: A charge starts as requires_signature — meaning the Solana transaction has been built but not yet signed and broadcast. After you sign with Phantom and broadcast, the settlement happens on-chain. The backend doesn't currently track post-broadcast status (it's a hackathon demo).
transaction_payloadstring +
Base64-encoded unsigned Solana transaction.
What to do with this: Decode from Base64 → create a Transaction object → refresh the recentBlockhash (it may expire) → sign with Phantom → broadcast to Devnet. The transaction contains a SOL transfer from your source_wallet to the developer's destination_wallet. See the signing code in Step 4 below.
exchange_rate_sol_usdnumber +
SOL/USD rate used at charge time from live oracle.
How conversion works: The backend fetches the live SOL price from CoinGecko at the moment you create the charge. amount_sol = amount_usd / exchange_rate_sol_usd. The USD cost is always fixed (set by category), but the SOL amount changes with the market. This rate is recorded for transparency.
Sign and broadcast the transaction

Decode the transaction_payloadA Base64-encoded unsigned Solana transaction. Contains a SOL transfer instruction with your wallet as feePayer. Decode it, refresh the blockhash, sign with Phantom, and broadcast to Devnet., refresh the blockhash, sign with Phantom, and broadcast:

JavaScriptimport { Transaction, Connection } from '@solana/web3.js';

const connection = new Connection(
  'https://api.devnet.solana.com', 'confirmed'
);

// 1. Decode the payload from the charge response
const tx = Transaction.from(
  Buffer.from(charge.transaction_payload, 'base64')
);

// 2. Refresh blockhash (prevents expiry)
const { blockhash } = await connection.getLatestBlockhash('confirmed');
tx.recentBlockhash = blockhash;

// 3. Sign with Phantom (popup appears)
const signed = await window.phantom.solana.signTransaction(tx);

// 4. Broadcast to Devnet
const signature = await connection.sendRawTransaction(
  signed.serialize()
);

console.log(`Settled: https://explorer.solana.com/tx/${signature}?cluster=devnet`);
Done! Payment settles on Solana Devnet in under 1 second. The developer receives their SOL, and you can verify the transaction on Solana Explorer.

Charges

Charges represent a payment for a registry service. Create one to get an unsigned Solana transaction, then sign and broadcast it.

Create a charge

POST/api/v1/chargesAuth required

Returns an unsigned Solana transaction for a registry service. The backend automatically fetches the live SOL/USD price, calculates the exact SOL amount, looks up the developer's wallet, and builds the transaction. You just sign it.

service_idstringrequired +
The registry service to pay for. Must exist in the Bazaar.
How to find valid IDs: Call GET /api/v1/registry/discover to browse all services. Use the id field from any service object. Prebuilt examples include: finance_stocks, weather_openmeteo, nlp_sentiment, search_web, openai_chat. If the ID doesn't exist, you get a 404 not_found_error. See the full catalog for all 26 IDs.
source_walletstringrequired +
Base58 Solana public key of the wallet paying.
Critical: This address is set as the feePayer on the Solana transaction. It must match the wallet you'll sign with in Phantom. If they don't match, the transaction will be rejected. Get your address from the Phantom extension. Example: AuofYo21iiX8NQtgWBXLRFMiWfv83z2CbnhPNen6WNt5
Idempotency-Keystringheader · optional +
Unique string preventing duplicate charges on retry.
Usage: Pass a UUID in the Idempotency-Key header. If the same key is sent twice, the second call returns the first call's cached response — no double-charge. Essential for production agents that retry on timeouts. Stored in-memory, resets on server restart.

Max records per page. Default: 10, max: 100.
Pagination: Use limit + offset together. Response includes has_more: true if more pages exist and total_count for total records. Example: ?limit=5&offset=10 returns records 11–15.

Records to skip. Default: 0.
Zero-indexed. Combine with limit to paginate through results. The response tells you has_more and total_count so you know when you've reached the end.

AI Gateway

The AI Gateway lets agents interact with the Bazaar using natural language. The AI decides whether to answer directly or request a paid tool call.

Chat completions

POST/api/v1/chat/completionsAuth required

Send a natural language message. The AI either answers directly (type: "message") or requests a paid tool call (type: "tool_call").

messagestringrequired +
The user's message to the AI.
What happens: The AI analyzes your message and decides if it can answer from its knowledge or if it needs live data from a Bazaar service. If it needs data, the response includes type: "tool_call" with the specific service_id and estimated cost. You then create a charge, pay, get the data, and send it back via /chat/tool-result.
conversation_idstringoptional +
Continue an existing conversation.
How conversations work: Each call returns a conversation_id. Pass it back in the next call to continue the conversation with full context. Omit to start a new conversation (one is auto-generated). Server stores up to 20 messages per conversation in memory. Resets on restart.
source_walletstringoptional +
Informational only — not used to build the transaction here.
Why it's optional: The chat endpoint doesn't create charges. It only tells you what service to pay for. The actual charge is created separately via POST /api/v1/charges where source_wallet is required.
Two response types: type: "message" = plain AI answer (no payment needed). type: "tool_call" = AI needs live data. Create a charge → sign → broadcast → send data back via /chat/tool-result.
Tool call response{
  "type": "tool_call",
  "conversation_id": "conv_a8f2k9x1b3",
  "message": "I need to access Stock Price Feed. This requires ~$0.01 in SOL.",
  "tool_call": {
    "id": "call_abc123",
    "service_id": "finance_stocks",
    "service_name": "Stock Price Feed",
    "estimated_cost_usd": 0.01
  }
}

Submit tool result

POST/api/v1/chat/tool-resultAuth required

After paying for a tool call, send the real data back. The AI generates a natural language answer.

conversation_idstringrequired +
The conversation_id from the tool_call response.
Must match exactly. This links the tool result to the original conversation context. The AI uses the conversation history to generate a contextual response. If mismatched, the AI won't have context about what data it asked for.
tool_resultstringrequired +
The raw data from the service. JSON-stringified or plain text.
What to send: The actual response from the paid service, as a string. If the service returned JSON, stringify it: JSON.stringify(serviceResponse). The AI will interpret and summarize the raw data into a natural language answer for the user.
service_idstringoptional +
Fallback tool call ID if conversation history is missing.
When needed: If for some reason the conversation context was lost (e.g. server restart), passing service_id helps the AI understand what tool was called. Under normal operation, the conversation_id carries this context.

Direct AI chat

POST/api/ai/chatPublic

Simple pass-through to OpenAI. No tools, no registry, no conversation history. One-shot message → reply.

messagestringrequired +
The prompt to send to OpenAI.
Behavior: Fires a real on-chain Devnet transaction (tiny amount) as proof-of-payment, then forwards your prompt to OpenAI. The response includes the AI reply, model used, token usage, and the Solana transaction signature as proof.
modelstringoptional +
OpenAI model. Default: gpt-4o-mini.
Available models: gpt-4o, gpt-4o-mini, o1-mini, o3-mini. If you pass an unsupported model name, it silently falls back to gpt-4o-mini.

Errors

All errors return a consistent JSON object with a machine-readable type and a human-readable message.

Error object

error.typestring +
Machine-readable error category.
Possible values:
invalid_request_error — missing or invalid parameters (400)
not_found_error — resource doesn't exist (404)
api_error — unexpected server failure (500)
gateway_error — upstream AI service unavailable (502)
error.messagestring +
Human-readable description. Safe to display to end users.
Usage: Show this directly in your UI or logs. Always a clear English sentence describing what went wrong and often hinting at what to fix. Example: "'service_id' is required and must be a non-empty string."

201Created — service successfully registered
400Bad Request — missing or invalid parameters
401Unauthorized — missing or invalid API key
404Not Found — service or charge doesn't exist
500Internal Server Error — unexpected failure
502Gateway Error — upstream AI service down, retry
